What should be Automated?

Let's talk about which components, functional and test cases should be covered by automation tests.

May 21, 2017 Sergey Larionov

Overview

To draw a plan of automation introduction into the project and to understand, what should be automated, we need to define the follow:

  • components, which the product consists of
  • set of necessary cases;
  • available cases (already written and used to test existed functionality);
  • frequency of functionality changing

After defining all the points, described above, we can determine the entry point to start the introduction of automated testing. Trying to automate all at once is a faulty action, as it can lead to unstable, continuously changing tests, and permanent changes in the automation framework. As a result, QA Automation Engineer will spend all his time on bonding and reworking tests.

Stages and Activities

Defining the product components

Usually (but not always) a product consists of at least backend/server and client side (web client, desktop, mobile client, etc.). First of all, it's necessary to distinguish the key components* of the product, and to determine, which of such components are least available for direct manual testing.

* Under key components we mean the parts, failure in which will impact on the efficiency of the product as a whole.

Thus, we'll distinguish those parts of the product, which testing needs to be automated at first, as using automated tests will reduce risks of releases with high priority bugs. Typically, backend is a component, which tests should reasonably be automated first of all, because:

  • it is the integral point of failure;
  • automated tests for backend are stable;
  • backend is least available for direct manual testing.
UI, Mobile client and other product components' tests should be automated after backend tests

Test cases for automation

The next step is to specify and distinguish test cases, which should be implemented to automated cases. If there is a prepared set of tests for existed functionality already, the queue and priorities of test cases should be defined. In case of new functionality, which is on the design stage, or untested functionality, a test plan should be prepared.

Test plan creation

Test plan includes the following steps:

  1. Specifying of the required behavior of functionality
    Functional behavior and basic use cases should be documented on the design stage by Project Manager.
  2. Analysis of interfaces and protocol of new functionality, created by Architects and Developers on the stage of DEV Design
  3. Thinking over of possible test methods, which can be useful for automated testing framework, and discussing them with Developers, if necessary.
  4. Development of test cases for new functionality.
  5. Division test cases on groups
    Tests should be divided the way to make it possible to understand, which sets of tests should be automated first of all. As each project and functionality has its own specifics, first of all you need to use common sense and understand, what role for the Team and product these tests will play, when determine the importance of automation of any test.

Tests can be prioritized by different parameters:

  • type of check;
  • test run frequency;
  • data, used by test;
  • amount of time on manual testing.

The prioritization, proposed below, is general and can be varied, depending on specifics:

  1. Acceptance tests - the first candidate on the automation, because they prove, that product component works through such protocol and interface, as other components of the product expect
  2. Priority regression tests
  3. Tests, that are hard to perform manually, and tests, taking a lot of time on manual testing
  4. Tests, that use large dataset
  5. Tests, using different dataset

The following groups of tests should be chosen for automation first of all

  1. tests, which will help the Team to make decisions before releases quicker;
  2. tests, that will help developers to change the functionality, using these autotests as a tool to verify its performance;
  3. tests, which will help to rid QA Engineers from routine activities, save the time and spend it on more useful things
It is important to understand, that robots don't make mistakes in contrast to people!
